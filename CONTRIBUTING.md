# Contributing to rsvelte

Thank you for considering a contribution! rsvelte is a Rust port of the
official Svelte compiler. Our goal is **bit-for-bit compatibility** with the JS
implementation, verified by a fixture-driven test suite.

This guide focuses on the pieces that aren't obvious from reading the code:
how the test suite is structured, how to run a single fixture, and how to
debug a failure. For the higher-level project goals, see [`AGENTS.md`](AGENTS.md)
(the same content is mirrored in `CLAUDE.md` for AI agents).

## Environment setup

### Native (recommended for daily work)

You need:
- Rust **1.90+** (see `Cargo.toml` `rust-version`). Use `rustup update stable`.
- Node.js **20+** and **pnpm 10+** (the repo's package scripts assume pnpm).
- Git with submodule support.

```bash
git clone <repo-url>
cd rsvelte
git submodule update --init --recursive
git config core.hooksPath .githooks   # enable cargo fmt/clippy pre-commit
pnpm install
pnpm run generate-fixtures            # required before tests
cargo test
```

The Svelte submodule is pinned. Do **not** update it casually — fixtures are
keyed on its commit hash and a wrong submodule version means tests compare
against the wrong expected output. Use the `upgrade-svelte` workflow instead.

### Docker (optional)

If you'd rather not install the toolchain locally:

```bash
docker compose up -d
docker compose exec dev bash
docker compose exec dev cargo test
```

The container provides a known-good Rust nightly + Node 22 + pnpm.

## Test suite at a glance

All compiler tests are integration tests under `tests/`. Each suite is its own
file (`tests/parser_fixtures.rs`, `tests/runtime.rs`, …) and uses
`tests/common/mod.rs` for shared utilities (path resolution, fixture loading,
canonicalization, diff output).

The data flow is:

```
svelte/ submodule (pinned commit)
  └─ packages/svelte/tests/<suite>/samples/<sample>/
        ├─ input.svelte       # source, written by Svelte authors
        ├─ _config.js         # test options (compile flags, expected errors)
        └─ expected files     # the JS suite's reference output
                ↓
pnpm run generate-fixtures
  └─ fixtures/<svelte-short-hash>/<suite>/<sample>/
        ├─ client.js          # JS compiler's output, stored verbatim
        ├─ server.js
        ├─ ast.json (parser suites)
        ├─ metadata.json
        └─ … (suite-specific files)
                ↓
cargo test
  └─ tests/<suite>.rs
        ├─ runs the Rust compiler on input.svelte
        ├─ canonicalizes via OXC parse→codegen (common::canonicalize_js)
        └─ compares against the stored fixture
                ↓
on failure → fixtures/.../_actual.<ext> is written for diffing
```

The current pass/fail breakdown lives in `AGENTS.md` and is regenerated by
`pnpm run compatibility-report`.

## Running tests

### Whole suite

```bash
cargo test                               # all tests
cargo test --release                     # recommended for full runs
cargo test --test parser_fixtures        # one suite
cargo test --test runtime -- --nocapture # with stdout
```

### A single fixture

`cargo test` filters by test function name, not fixture name, so it can't
target one sample directly. Use the helper:

```bash
scripts/dev/test-sample.sh runtime-runes await-block-state
scripts/dev/test-sample.sh ssr if-block
scripts/dev/test-sample.sh compiler-errors duplicate-prop
```

The helper regenerates that one fixture, sets `DEBUG_TEST=<sample>` so the
runner prints a canonical diff on mismatch, and runs the suite single-threaded.

### Manual debug env vars

These are read by `tests/common/mod.rs::compare_js_with_debug` and apply to
suites that delegate through it (currently `tests/runtime.rs`):

| Var                 | Effect                                                   |
|---------------------|----------------------------------------------------------|
| `DEBUG_TEST=<name>` | Print canonical expected/actual diff for the named test  |
| `DEBUG_ALL=1`       | Print canonical diffs for every failing test             |
| `DEBUG_RAW=<name>`  | Also write raw + canonical inputs to `/tmp/debug_*.js`   |
| `WRITE_ACTUAL_OUTPUT=1` | Write `_actual.<ext>` next to every fixture even on pass |

## Adding or fixing a test

We do **not** add bespoke tests — every fixture must exist in the upstream
Svelte test suite first. This keeps "compatibility" honest.

### Scenario A: a Svelte fixture exists, we just don't pass it

1. Confirm the fixture is in the submodule:
   ```bash
   ls submodules/svelte/packages/svelte/tests/<suite>/samples/<your-sample>
   ```
2. Make sure your local fixtures are fresh:
   ```bash
   pnpm run generate-fixtures -- --category=<suite> --sample=<your-sample> --force
   ```
3. Iterate with `scripts/dev/test-sample.sh <suite> <your-sample>`.
4. Fix the Rust code under `src/compiler/...` until canonical output matches.
5. Run the full suite once before pushing:
   ```bash
   cargo test --release
   ```

### Scenario B: you want a fixture that doesn't exist upstream

Send a PR to <https://github.com/sveltejs/svelte> first. Once it's merged we
bump the submodule, regenerate fixtures, and the new sample shows up here
automatically.

### Scenario C: you found a divergence the JS compiler also has

File an issue upstream. Don't paper over it locally — `compare_js` would then
silently agree with broken output forever.

## Debugging a fixture failure

1. **Identify the suite and sample.** `cargo test` prints them in the failure
   summary. CI also uploads a `test-failures-<os>` artifact containing the
   `_actual.*` files for every diverging fixture.

2. **Look at the canonical diff.**
   ```bash
   DEBUG_TEST=<sample> cargo test --test runtime -- --nocapture
   ```
   The `+` lines are what we produced; `-` lines are what we should have
   produced. Both sides are already canonicalized, so any difference is a real
   semantic divergence — not a whitespace artifact.

3. **Look at raw outputs if the canonical diff is unhelpful.**
   ```bash
   DEBUG_RAW=<sample> cargo test --test runtime -- --nocapture
   ls /tmp/debug_raw_*.js /tmp/debug_canonical_*.js
   ```

4. **Trace through the relevant compiler phase.** The phases are:
   - `src/compiler/phases/1_parse/` — Svelte → AST
   - `src/compiler/phases/2_analyze/` — scope, runes, bindings
   - `src/compiler/phases/3_transform/` — AST → JS (client/server)

   Start by narrowing the diff to a single output line, then `grep` the
   surrounding identifier in the transform phase.

5. **Don't update the fixture to match.** Fixtures are regenerated from the JS
   compiler — editing them by hand desynchronizes the source of truth. The
   only legitimate way to change a fixture is to bump the Svelte submodule.

## Performance work

See [`docs/perf-roadmap.md`](docs/perf-roadmap.md) for the long-running plan. Quick refs:

```bash
scripts/bench/bench.sh --quick          # ~30s comparison vs official compiler
scripts/bench/bench.sh --criterion      # full criterion run (HTML reports under target/criterion/)
scripts/bench/bench.sh --profile path/to/component.svelte
```

Benchmarks are not yet enforced in CI, so any performance-sensitive change
should be validated locally and the result mentioned in the PR description.

## Commit and PR conventions

- Run `cargo fmt` and `cargo clippy --all-targets --all-features -- -D warnings`
  before committing. The pre-commit hook does this automatically once you've
  enabled `core.hooksPath`.
- Atomic commits — one logical change per commit.
- Don't bypass hooks (`--no-verify`) unless you're explicitly asked to.
- PRs to `main` go through CI: format, clippy, all-OS tests, compatibility
  report. The compatibility-report job posts a delta comment on the PR; if it
  shows regressions, treat that as a blocker.
- Don't manually regenerate `apps/playground/static/test-results.json` from a feature
  branch — `pnpm run test-and-update` is for the maintainer who lands on
  `main` after a Svelte bump.
